<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent → Noren (HTTPS) | First outbound network call. Serial (semi-trusted local input) leaves the machine; agentToken (bearer secret) enters. |
| local user → serial field | Serial is local input, but the authoritative trust decision is made server-side by Noren. |
| cargo → crates.io | Supply-chain: new `winreg` dep + first activation of `printers`/`auto-launch`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Noren endpoint (fake server MITM) | mitigate | `reqwest` with `rustls` validates TLS against system/webpki roots; `noren_base_url()` must resolve to an `https://` URL (verify default constant uses https). |
| T-02-02 | Information Disclosure | agentToken in ActivateResponse | mitigate | Token is returned in-memory only; noren_client never logs the response body; token is NOT persisted by this plan (Plan 03 persists via DPAPI). |
| T-02-03 | Tampering | new/newly-activated crates (winreg, printers, auto-launch) | mitigate | Blocking human-verify checkpoint (Task 3) confirms legitimacy on crates.io before build; T-02-SC below. |
| T-02-04 | Information Disclosure | serial value in logs | accept | Serial is not a long-lived secret (validated server-side, per-machine); do not log it, but no encryption required. |
| T-02-SC | Tampering | cargo installs (winreg + [ASSUMED] deps) | mitigate | slopcheck unavailable at research time → all [ASSUMED]; blocking human checkpoint required before their first build (non-auto-approvable). |
