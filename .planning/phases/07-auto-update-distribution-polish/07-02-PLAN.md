<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| S3/CDN feed → Velopack UpdateManager | The `.nupkg` + `releases.win.json` served at `feed_base_url` are untrusted until SHA256-verified |
| Background Tokio task → event-loop thread | Only `UserEvent::UpdateStaged` crosses (C2: no direct tray mutation off-thread) |
| Staged update → next launch | Velopack updater applies the staged package; must be the version-gated, integrity-checked one |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-tamper | Tampering | Velopack apply path (apply.rs) | mitigate | `stage_update` is reached only after plan-01 `verify_sha256` passes (Task 2/3 ordering); Task 4 SC-2 proves the mismatch abort never applies |
| T-7-MITM | Tampering / Disclosure | `downloadUrl` transport | mitigate | reqwest+rustls HTTPS with TLS validation on by default (existing shared Client) — no plaintext download |
| T-7-replay | Elevation | applying an old/duplicate update | mitigate | `check_for_update` semver ordering gate (plan 01) + Velopack's own `check_for_updates` version compare — both require remote > current |
| T-7-C2 | Tampering | tray mutation from background thread | mitigate | Tray updates ONLY via `UserEvent::UpdateStaged` on the event-loop thread (Pattern 6 / pitfall C2) |
| T-7-2xtoast | (UX integrity) | double-toast on 6h re-poll | mitigate | `update_staged` once-per-session gate in `run_update_check_loop` (Task 3) — asserted in Task 4 SC-1 |
| T-7-V2 | Info Disclosure | agent_token in update error path | mitigate | Token only via `.bearer_auth()` in `check_version`; update task `eprintln!`s never include it (T-02-02) |
| T-{phase}-SC | Tampering | npm/pip/cargo installs | accept | No new crates — `velopack = "1"`, `tauri-winrt-notification = "0.8"` already present under `[target.'cfg(windows)'.dependencies]`; no install step in this plan |
</threat_model>
