<objective>
Stand up the cross-platform Cargo foundation for Brevly Print: a `lib`+`bin` project whose
**portable core compiles and tests on Linux** and whose **full v1 dependency set compiles on
Windows** (Windows-only crates isolated under `[target.'cfg(windows)'.dependencies]` +
`#[cfg(windows)]`). This plan delivers everything below the event loop: the manifest, the
app-dir initializer, the module skeletons for the config + credential stores (with the
`CredentialStore` trait and its two `cfg`-gated impls), the Wave-0 integration-test
scaffolds, and the ubuntu+windows CI matrix.

Purpose: SC-4 (compiles on both targets) is the gate every other Phase 1 plan builds on.
Establishing the cross-platform structure here means plans 01-02 and 01-03 never re-litigate
target gating — they fill contracts this plan defines.

Output: A `cargo build` that exits 0 on Linux and (on Windows CI) links the full v1 set; an
idempotent app-dir initializer; compiling stubs for the two stores; and the CI matrix.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local disk ↔ app | `credential.bin` written/read from the per-user app dir; no network, no external input in Phase 1 |
| Windows user session ↔ DPAPI | `Scope::User` ties encryption to the Windows account (Windows only) |

**Note:** Phase 1 has essentially no external attack surface — no network, no user-supplied
input (the credential value is a dummy until Phase 2 activation). The only security-relevant
surface is the credential store, addressed below.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure / DoS | credential store (`credential.bin`) | mitigate | Windows: `windows-dpapi` `Scope::User` ties the blob to the user account (other users cannot decrypt); DPAPI key loss on reinstall (M7) surfaces as typed `CredentialError::Corrupt`, never a panic → Phase 2 re-activation. Contract tested on Linux (DevFile), real DPAPI path tested on Windows. |
| T-1-02 | Information Disclosure | Linux `DevFileCredentialStore` | accept (non-shipping) | Plaintext file is DEV/TEST ONLY and MUST NEVER ship — enforced by `#[cfg(not(windows))]` gating + explicit doc-comment. Product v1 is Windows-only (D-24). This is a documented non-shipping constraint, not a vuln to fix. |
| T-1-SC | Tampering | cargo/crates.io installs | mitigate | All crates VERIFIED on crates.io in RESEARCH §Package Legitimacy Audit (all `[VERIFIED: crates.io]`, none `[SLOP]`/`[SUS]`) — no blocking human checkpoint needed. |
</threat_model>
